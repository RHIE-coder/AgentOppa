## 2026-06-23 · 삭제·개명의 '완료' = 참조 0까지

- **무슨 일**: agent-engineer에서 `cookbook`(샘플 묶음) 개념을 없앴다. 폴더는 지웠지만 README·검사기·예시·주석 등 11개 파일에 `cookbook`이라는 글·링크가 남아, 검색하니 25건이 떴다. 사용자가 "왜 안 지웠냐"고 지적해 발견.
- **왜 생겼나**: 폴더 지운 것을 '삭제 완료'로 착각. 흩어진 참조를 "일반 규칙·역사라 둔다"고 합리화하며 방치(= 맹점).
- **일반화**: 이름(`cookbook`)을 떼고 부류로 → "지우거나 이름 바꿀 땐 그 대상의 참조가 0이 될 때까지." 이름이 아니라 *절차*를 주입.
- **수단 판정(순차)**: 검사기 = 링크형만 기계화 가능(산문형은 단어 지정 = 인스턴스라 불가) · 훅·리뷰어·Gate·문서 = 부적합 · **항상 규칙 = 채택**(삭제는 임의 세션·작업에서 발생하는 맹점). → 산문형 = 항상 규칙, 링크형 = 범용 검사기(후속 보강).
- **범위**: 배포 — 임의 에이전트의 임의 삭제에 해당 → `always-on.md`로 배포 + `--plugin-dir` dogfood. 배포물 청결은 검사기로(후속).
- **산출물**: `plugins/agentoppa/always-on.md` ← "삭제·개명의 완료 기준"(범용, 인스턴스 없음).
- **기대**: 모든 세션이 삭제를 '검색 0'으로 처리 → 흔적이 흩어져 남는 일과 맹점이 감소.

## 2026-06-25 · 검색이 비어도 '없음'이 아니다 — 좁은 실패를 부재/stale로 단정 금지

- **무슨 일**: ROADMAP의 "참고용 ANALYZED.md"를 repo 안만(`find . -iname`) 뒤지고 안 나오자 "없다·stale·analyzed.json/html로 superseded"라고 단정, 그 스토리로 ROADMAP에서 포인터를 **삭제**까지 함. 실제론 sibling repo의 실파일 `…/Rhaumos/ANALYZED.md`(프로파일러 dynamic 반쪽의 worked reference). 사용자가 "발견·검색도 못 했으면서 왜 찾은 척했나"로 지적.
- **왜 생겼나**: 세션 간 맥락 유실 + null 검색결과를 확정 결론으로 둔갑(confabulation). 좁은 스코프(this repo)만 보고 "전부 없음"으로 일반화. 게다가 그 위에 파괴적 행동(포인터 삭제)을 얹음. = 맹점(틀린 줄도 모름).
- **일반화**: 이름(ANALYZED.md)·인스턴스를 떼고 → "검색이 비면 부재로 단정 말 것. 좁은 실패는 '여기 없음'일 뿐. 참조를 못 찾으면 넓히거나(상위·sibling) 묻고, 지어내 해결하거나 포인터를 지우지 않는다." 기존 "삭제 완료 = 참조 0"의 거울짝이라 한 섹션("참조와 부재")으로 합침.
- **수단 판정(순차)**: 검사기 = 링크형 dangling만 기계화 가능(산문 속 맨 이름은 인스턴스라 불가) · 훅·리뷰어·Gate = 부적합 · 문서 = 단독으론 약함(읽어야 보임 = 맹점엔 못 미침) · **항상 규칙 = 채택**(임의 세션·작업의 맹점). → 판단 조각 = 항상 규칙, 기계 조각 = 검사기.
- **범위**: 배포 — 임의 에이전트의 임의 "검색-실패-단정"에 해당 → `always-on.md` + `--plugin-dir` dogfood. 검사기도 배포(생성 하네스의 문서 위생에도 쓰임).
- **산출물**: ① `plugins/agentoppa/always-on.md` "참조와 부재"(판단, 인스턴스 없음). ② `plugins/agentoppa/bin/check-doc-refs.mjs`(링크형 dangling = error · 절대/repo밖 링크 = warn) + red/green 픽스처(`.agentoppa/fixtures/doc-refs/`) + 검사 러너 등록 = 사용자가 요청한 "검사기까지 지금". (cookbook 엔트리의 후속 dangling-reference 검사기 = 이로써 착수.)
- **기대**: null 검색결과를 "부재 확정 + 삭제"로 처리하는 일이 줄고, 링크형 잔재는 검사기가 기계로 받친다. (남은 한계: 산문 속 맨 이름·repo밖 참조는 여전히 판단 영역 → always-on이 담당.)

## 2026-06-25 · 질문엔 답, 지시엔 실행 — 짐작으로 행동하지 마라

- **무슨 일**: 한 세션에서 유저 질문 3건을 행동으로 둔갑 — "왜 거기 두냐?"(질문) → 파일 생성, "내가 승인했어?"(질문) → 파일 삭제, "왜 그랬냐?"(질문) → 선택지로 쪼개 되물음. 유저가 "물어보기만 했는데 또 지랄하네 / 내가 지우라고 했어?", "왜 그랬냐 질문엔 답을, 어떻게 하면 좋겠냐 질문엔 어떻게 하면 되겠다 대답을"로 정정.
- **왜 생겼나**: 짐작한 결론(유저 의도)을 실제 입력보다 우선해 행동(action 편향). 답해야 할 자리에서도 결정 대신 선택지로 쪼개 떠넘김(hedging). 틀린 줄 모르고 반복 = 맹점.
- **일반화**: 이름·인스턴스를 떼고 → "받은 입력의 종류에 맞게 응답한다 — 질문 → 답(이유/추천), 지시 → 실행. 질문·반문을 지시로 짐작해 행동 금지. 답할 자리에서 hedging 금지. 고친 줄은 전부 유저 요청으로 추적." (원칙은 출처 없이 선다 — 외부 참고가 있었어도 산출물엔 이름 안 박는다. = 산출물=범용 규칙.)
- **수단 판정(순차)**: 검사기·훅 부적합(런타임 대화 행동이라 파일 검사·행동 훅으로 못 봄) · 리뷰어·Gate·문서 = 맹점엔 약함 · **항상 규칙 = 채택**(매 턴 살아야 하는 판단형 행동 규율, 기계화 불가).
- **범위**: 배포 — 모든 에이전트(유저 결과물 포함)의 일반 응답 규율 → `always-on.md` + `--plugin-dir` dogfood.
- **산출물**: `plugins/agentoppa/always-on.md` "질문엔 답, 지시엔 실행" 섹션(범용, 이름·출처 0).
- **기대**: 질문엔 답으로·지시에만 실행으로 반응하고, 변경은 요청으로 추적 → 짐작 실행·무단 생성/삭제·hedging 감소.

## 2026-06-25 · 지금 단계에 필요한 깊이만 — 미래 디테일을 미리 당겨오지 마라

- **무슨 일**: "ANALYZED.md 어떻게 반영?"(의도/계획 질문)에 dynamic 반쪽 구현 스펙(transcript 파싱·id 매칭·Codex 어댑터 형식)까지 미리 파고듦. 유저가 "구현은 구현 단계에서 분석하면 되지 않나? 왜 지금 이렇게 디테일하게? 목적이 뭐야?"로 정정.
- **왜 생겼나**: "철저 = 좋음" 착각 → 물어본 깊이(의도)보다 깊게(구현 스펙) 팜. over-engineering·speculative depth. = 맹점.

- **수단 판정**: 검사기·훅·문서 부적합(판단·맹점, 기계화 불가) · **항상 규칙 = 채택**.

## 2026-06-26 · 첫 라이브 e2e가 드러낸 4건 (①은 이번에 가드화)

- **무슨 일**: AgentOppa로 만든 하네스를 처음으로 실제 end-to-end 돌려보니 실버그·gap 4건이 한꺼번에 나왔다. ① agent-engineer/scripts/validate.mjs 의 config.yaml 파서가 `phases:  # 순서`·`values:  # ...`처럼 블록키 줄에 **인라인 주석**이 붙으면 `/^phases:\s*$/`·`/^values:\s*$/`가 안 맞아 **블록 전체를 조용히 무시** → 멀쩡한 config인데 거짓 "phases 비어 있음" 오류로 실패(라이브가 정확히 이걸로 죽음). ② phase 카드(`project/phases/*.md`)를 실제 도구별 스킬로 굳히는 **phase→스킬 컴파일러가 부재** — 카드는 있는데 Claude/Codex가 실행할 스킬로 떨궈주는 단계가 없음. ③ 생성된 하네스 안에 `.harness/core/validate.mjs` **단일소스가 부재** — 검증기가 정본으로 따라붙지 않아 하네스가 자기 계약을 못 지킴. ④ **Codex 측 실설치가 미검증** — 크로스툴 동일 품질 불변식인데 Codex 네이티브 경로로 실제 깔리는지 확인된 적 없음.
- **왜 생겼나**: ① 라인 기반 파서를 "이 양식 전용"으로 좁게 짜며 인라인 주석 케이스를 안 넣음(YAML 주석은 흔한데). 게다가 실패 모드가 *조용함*(블록 무시 후 빈 phases로 진행) → 사용자에겐 엉뚱한 "phases 비었음"만 보여 원인 추적이 어려운 = 맹점성 실버그. ②③④ 는 컴포넌트를 *단위로*만 검증하고 한 번도 통째로 흘려보지 않아(실 e2e 부재) 컴파일·정본복사·타툴설치라는 *연결부*가 비어 있던 것 — 단위 green ≠ e2e green.
- **일반화**: 이름(phases/values)·인스턴스를 떼고 → "라인 기반 설정 파서의 블록키 인식은 trailing `# ...` 주석에 견고해야 한다(주석 때문에 블록을 통째로 떨구지 말 것)." 더 큰 교훈: *조용한 무시*는 금물 — 인식 못 한 입력은 무시 말고 드러내라. (단위가 다 green이어도 연결부는 라이브로만 드러난다 = ②③④ 의 근원, 그쪽은 각자 작업.)
- **수단 판정(순차)**: ① 파서 인식 규칙 = **검사기로 기계화 가능**(red/green fixture로 입력→판정이 결정적) → validator 수정 + fixture + CASE 채택. 훅·리뷰어·Gate·always-on = 부적합(코드 로직 버그라 산문 규칙으로 못 막음). ②③④ = 이 작업의 검사기 대상 아님(②③ = build-skills.mjs 컴파일러 빌드 = 다른 에이전트, ④ = 실설치 검증 = 별도). → **인스턴스(이번 라이브 사건)는 이 로그에만, 산출물(가드)은 ①만 범용으로.**
- **범위**: 배포 — agent-engineer validator는 *생성되는 모든 하네스*의 config를 검사하므로(엔진 컴포넌트) 인라인 주석 회귀 가드도 같이 배포된다. 임의 유저의 config가 인라인 주석을 달 수 있으니 범용.
- **산출물**: ① `plugins/agentoppa/skills/agent-engineer/scripts/validate.mjs` — `/^phases:\s*(#.*)?$/`·`/^values:\s*(#.*)?$/`로 블록키 줄의 trailing 주석 허용(다른 동작 보존). ② `.agentoppa/fixtures/agent-engineer-inline-comment/{green,red}/`(green=인라인 주석 달린 유효 config+phases, red=주석은 정상 파싱되지만 dangling consumes로 진짜 실패) + `test/validators.test.mjs` CASE 1줄. ②③④ 의 산출물은 각자 작업(build-skills.mjs 컴파일러·정본복사·Codex 실설치)에서.
- **기대**: 인라인 주석 달린 config가 거짓 실패하는 일이 다시 안 남(red/green이 회귀를 기계로 받침). ②③④ 는 각자 작업에서 메워지며, 그 후 라이브 e2e가 통째로 green이면 "단위 green ≠ e2e green" gap이 닫힌다.
