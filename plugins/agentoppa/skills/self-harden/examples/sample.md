# 견본 — self-harden 출력 한 판 (목표 형식)

실제 출력이 이렇게 나온다. 제목·톤·*산출물/로그 분리*의 목표로 삼는다.

---

```
🔧 self-harden · 엔진은 남의 disposable 샘플을 참조하지 않는다
   엔진 스킬이 '언제든 빼도 되는' 샘플(다른 스킬의 examples/)을 링크하면 안 된다.

[1] 무슨 일이 있었나
    엔진 스킬(intent-interview)이 다른 스킬의 examples/ 파일을 직접 링크했다.
    사용자가 "엔진이 언제든 빼도 되는 샘플을 참조하면 안 된다"고 정정.

[2] 같은 종류 전체로
    엔진·프로세스 컴포넌트는 disposable 샘플(examples/)을 참조하지 않는다.
    └ 특정 스킬 이름은 부류에서 뺀다 → 로그로.

[3] 어떻게 막을까 — 강한 것부터 순차
    검사기    ✓  링크는 '경로'라 기계로 잡힘 → 채택(스킬 글에서 남의 examples/ 링크 탐지)
    훅/리뷰어  ✗  특정 행동·리뷰 순간이 아님
    Gate/문서  ✗  붙일 단계 없음 / 읽어야 보임(맹점에 약함)
    항상 규칙  ✓  글로만 언급한 형태는 기계로 못 잡는 맹점 → 채택
    └ 쪼갬: 링크형 = 검사기 / 산문형 = 항상 규칙

[4] 누구에게 적용 : 배포 (생성되는 하네스에도 같은 구조가 생기므로 유저에도)

[5] 어디에 넣나
    산출물 → ccc-plugin 검사기(스킬 글 훑기) + always-on.md 규칙   «범용, 스킬 이름 없음»
    로그   → .agentoppa/hardening-log.md

[6] 로그 전문 (hardening-log.md)
    ## <날짜> · 엔진 ↛ disposable 샘플
    무슨 일    intent-interview가 다른 스킬의 examples/ 파일을 링크.
    왜 생겼나  '재사용 샘플이니 참조해도 된다'고 착각(= 맹점).
    일반화     엔진은 disposable 샘플(examples/) 참조 안 함. (스킬 이름은 뺌.)
    어떻게 막기 링크형=검사기(빨강/초록) · 산문형=항상 규칙(맹점이라 자동검사 어려움).
    어디에     ccc-plugin/scripts/validate.mjs + always-on.md
    기대       엔진이 샘플에 의존하면 즉시 빨강 → 샘플을 통째로 빼도 프레임워크 멀쩡.

[7] 이대로 진행할까요?    [진행] / [고칠 점]
```

## 왜 좋은 견본인가
- **제목**이 도구 정의("실수 안 하게")가 아니라 *이 교훈의 한 줄 요약*.
- 인스턴스(스킬 이름)는 *산출물*에 안 들어가고 *로그*에만 — 산출물은 라이브러리처럼 범용.
- 한 교훈을 *수단 순차 판정*으로 쪼갬 — 링크형=검사기, 산문형=항상 규칙.
- 가드를 *말*이 아니라 빨강/초록 *테스트*로.
